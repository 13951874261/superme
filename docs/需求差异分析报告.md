# 项目代码与需求差异分析报告

> 基于 `mt.md` 需求文档与 `src/` 及 `vocab-server/` 代码库的完整比对分析
>
> 分析时间：2026-05-30

---

## 整体评估

项目基于 `mt.md` 的八大核心训练系统框架，已实现了**约 65-70%** 的核心功能落地。主要强项集中在英语引擎（词汇、口语、写作、听力）和驭心博弈系统，界面也基本遵循了 70/30 折叠布局和控制论闭环的设计理念。但需求文档中有大量精细化机制和模块在代码中尚未完整实现或完全缺失。

---

## 一、功能遗漏（完全未实现）

### 1. 【洞察系统】三段论 / 归纳演绎每日训练实例

**需求描述：**
每日推送三段论、归纳演绎及非形式谬误（如以偏概全、滑坡谬误）的训练实例，强制要求用户在复杂信息中精准抓取逻辑破绽、事实破绽与意图破绽。

**现状：**
`ListenModule.tsx`（洞察系统）仅有按"体制内/外企/通用逻辑"分类获取场景题目的功能，调用 `fetchDynamicInsightScenario` 从 Dify 获取。没有独立的每日三段论/归纳演绎/非形式谬误专项训练模块，也没有自动推送机制。

**代码位置：** `src/components/modules/ListenModule.tsx`

**提升建议：**
新增独立"逻辑破绽识别"模块（Tab），内置非形式谬误分类库（三段论、归纳演绎、以偏概全、滑坡谬误、稻草人谬误等），每日自动推送训练题，用户提交后 AI 评判其破绽识别准确率并记录到训练数据中。

---

### 2. 【洞察系统】弦外之音与心理侧写——三维场景完整覆盖

**需求描述：**
覆盖体制内（汇报、晋升）、跨国企业（绩效反馈、跨文化交流）及社交场景，强制要求分析发话者的社会层级、内在水准、利益诉求与弦外之音。

**现状：**
`ListenModule.tsx` 的分类仅有"体制内/外企/通用逻辑"三档，没有社交场景分类，且没有独立的"心理侧写"分析维度输入/输出。场景题目的结构化维度（社会层级/内在水准/利益诉求）未在 UI 上体现为必填字段。

**代码位置：** `src/components/modules/ListenModule.tsx`

**提升建议：**
在洞察系统增加场景类型选择器（四分类：体制内汇报/跨国绩效反馈/跨文化交流/社交场景），每个场景对应独立 prompt 模板和评判维度。输入区应强制用户填写"发话者身份层级"和"利益诉求推测"，输出区结构化呈现"弦外之音/社会层级/内在水准/利益诉求"四维分析。

### 3. 【破局系统】金字塔 / 因果 / 对比等表达模板体系

**需求描述：**
提供金字塔、因果、对比等表达模板。要求用户根据不同权力角色（向上汇报、平级协调、向下布置、对外公关）调整称呼、语气和内容边界，进行口播练习。

**现状：**
`SpeakModule.tsx`（破局系统）有"结构化表达/精准提问/即兴反击"三种训练模式，但没有任何预置的表达模板（金字塔/因果/对比），也没有按"权力角色"筛选或生成对应话术的功能。

**代码位置：** `src/components/modules/SpeakModule.tsx`

**提升建议：**
新设表达模板库，每种模板（金字塔、STAR、PREP、利弊分析等）配合权力角色选择，AI 自动生成符合对应场景的练习题目。

### 4. 【英语引擎】严格主题通关机制——邮件写作第三项硬指标缺失

**需求描述：**
每个商务主题须达成三大硬指标：① 5 分钟脱稿发言；② 与 AI 对话 10 轮；③ 撰写 1 篇无错漏邮件。

**现状：**
`EnglishContext.tsx` 的通关判断标准仅为 `oralCount >= 10 && maxWriteScore >= 8`（口语 10 轮 + 写作 8 分），`impromptuPassed`（即兴演讲通过）也是独立条件。**邮件写作指标（第三项硬指标）完全缺失**——没有检测用户是否完成了"1 篇无错漏邮件"的验证逻辑。

**代码位置：** `src/components/modules/english/context/EnglishContext.tsx`

**提升建议：**
在 `EnglishContext.tsx` 的 `isMastered` 判断逻辑中增加第三项：`emailCompleted: boolean`。在 `WriteTab.tsx` 或独立 Tab 中，当用户提交一封邮件且 Dify 批阅 L1（语法）达到满分时，触发 `emailCompleted = true`。通关条件变为三项全满足。

### 5. 【英语引擎】找破绽与精准反击——每日专属词汇推送与场景埋设漏洞

**需求描述：**
每日专属推送"识别破绽"的词汇（如 contradiction, ambiguity）与句型。在口语与听力模拟中，AI 刻意埋设逻辑漏洞，要求用户用英语指出破绽并设计兼顾商务分寸的精准提问。

**现状：**
`OralWarRoom.tsx` 的场景对话中有 `flaw_point` 反馈，`ListenTab.tsx` 有"截获黑话"功能，但没有独立的"每日破绽词汇推送"机制（每天推送 contradiction、ambiguity、fallacy、syllogism 等词汇及其商务场景应用句型），也没有在"精听"和"口语沙盘"中刻意埋设逻辑漏洞供用户发现并用英语反击的功能。

**代码位置：** `src/components/modules/OralWarRoom.tsx`、`src/components/modules/english/tabs/ListenTab.tsx`

**提升建议：**
新增"每日破绽词汇推送"组件（可集成在 `DashboardTab`），自动推送 5-8 个"识别破绽"主题词汇及对应商务句型。在 `OralWarRoom.tsx` 的场景生成逻辑中增加隐藏参数，让 AI 在特定回合对话中植入逻辑漏洞（如数据矛盾、因果倒置、以偏概全），用户识别并用英语反击后给予额外分数奖励。

### 8. 【私人认知树洞】真正的动态进化反馈

**需求描述：**
用户每周输入实战心得、深度阅读感悟或职场困境，AI 据此动态调整下周的人性博弈、英语和认知升维推送方向，实现系统的动态进化。

**现状：**
`WeeklyChatModule.tsx` 仅有静态 UI，有一个"AI 进化反馈"的假数据区域，没有任何真实的 AI 调用逻辑（没有 Dify 对接、没有上下文注入、没有动态推送调整机制）。此外，文本框（textarea）没有绑定任何 value 和 onChange，用户甚至无法输入并提交任何内容。

**现状：**
`ReadModule.tsx` 的"财报研判"场景中有 `traceability_training` 输出项，"外企邮件"场景中有 `stance_reversal` 输出项——这些仅是被动展示的 AI 分析结果。用户缺少主动练习这些技能的交互闭环（用户主动选择立场并撰写反向解读，或主动输入溯源路径）。

**代码位置：** `src/components/modules/ReadModule.tsx`

**提升建议：**
在 `ReadModule.tsx` 的外企邮件场景中，将"立场反转练习"从被动展示改为独立交互流程：用户先读原始邮件 → 选择一个利益立场 → 用英语/中文撰写反向解读 → AI 评判立场切换的准确度和逻辑严密性。信息溯源训练类似：用户看到一个数据/结论，需主动输入"该数据的可能来源"和"验证路径"，AI 评判其溯源意识。

### 10. 【驭心博弈系统】顶层认知升维——独立训练模块

**需求描述：**
跳出做事层，通过大历史观、权力结构、周期规律与第一性原理，进行 5-10 层的长期因果链推演，完成战略级决策者的思维跃迁。

**现状：**
`GameTheoryModule.tsx` 的报告区域中有 `causal_chain`（10 重因果链条）展示，但这是在博弈案例分析后被动输出的结果，不是独立的"顶层认知升维训练模块"。没有独立的入口让用户主动进行大历史观/权力结构/第一性原理的练习。

**代码位置：** `src/components/modules/GameTheoryModule.tsx`

**提升建议：**
新增独立 `StrategicMindModule.tsx`（顶层认知）。用户输入一个商业/政治/历史事件，通过 Dify 工作流自动生成从"表面事件→权力结构→历史周期→第一性原理"的多层推演图谱（可用思维导图或时间轴可视化），并要求用户撰写自己的推演路径，AI 比对评判。

### 11. 【后端引擎架构】Dify v0.6+ Conversation Variables 与跨会话弱点追踪

**需求描述：**
Dify 后端系统应支持持久化跨会话变量（Conversation Variables），实现对用户能力的长期画像跟踪与智能调控。

**现状：**
当前的 Dify API 请求封装（`difyAPI.ts` 及各模块 fetch 调用）中，所有会话都是完全无状态或纯单次会话，未包含对 `conversation_variables`（如 `User_Current_Profile`）的维护与回传。AI 引擎无法在多次对话间真正沉淀并调用用户先前累积的认知弱点。

**代码位置：** `src/services/difyAPI.ts`

**提升建议：**
重构接口层，在 API 交互请求体中带入 `conversation_variables` 结构体，支持拉取和同步 Dify 后端的云端持久化用户字典变量。

## 二、功能提升（已部分实现，需深化）

### 1. 【英语引擎】每日 50 词汇 + 30 短语推送量控制

**需求描述：**
每日推送 50 个主题词汇（带发音）、30 个短语（5 个例句），严格控制每日配额，未达上限前每次触发从 Dify 提纯结果中截取对应数量写入生词本。

**现状 vs 需求：**
2026-05-30 已完成深化实现。

**已实现功能：**
- `vocab-server/server.js` 新增 `daily_vocab_quota` 表（`user_id + quota_date` 联合唯一键），记录每日词汇/短语入库数量
- 新增 `POST /api/english/daily-extract` 接口：调用 Dify 工作流提纯词汇 → 检查今日配额（50 词 / 30 短语）→ 严格按剩余配额截取入库 → 更新配额记录
- 新增 `GET /api/daily-quota/status` 接口：查询当前配额状态
- `difyAPI.ts`：`triggerEnglishMasteryExtraction` 改为调用新端点，支持 `quotaExceeded` 字段判断配额耗尽
- `DashboardTab.tsx`：弹药补给库区域新增**实时配额进度条**（词汇/短语各自显示已用/剩余），提取按钮根据配额耗尽状态显示差异化提示

**配额耗尽处理：**
- 配额用尽后再次触发，`quotaExceeded = true`，前端显示"今日配额已用尽，明日再来"
- 配额进度条耗尽时显示红色，直观提示用户

**代码位置：**
`vocab-server/server.js`（后端配额逻辑）、`src/services/difyAPI.ts`（API 封装）、`src/components/modules/english/tabs/DashboardTab.tsx`（前端配额展示）

**提升建议（后续可选）：**
- 在每日配额之外增加"紧急补充"通道（需手动确认放弃艾宾浩斯曲线复习节奏）
- 配额耗尽时推荐用户使用划线制卡作为补充弹药来源
